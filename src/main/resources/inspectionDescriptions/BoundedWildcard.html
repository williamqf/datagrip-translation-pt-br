Relata parâmetros de métodos genéricos que podem usar wildcards limitados (bounded wildcards).

Exemplo:


  void process(Consumer<Number> consumer);

Deve ser substituído por:


  void process(Consumer<? super Number> consumer);

Esta assinatura de método é mais flexível porque aceita mais tipos: não apenas Consumer<Number>, mas também Consumer<Object>.

Da mesma forma, para parâmetros de tipo em uma posição covariante:


  T produce(Producer<T> p);

Deve ser substituído por:


  T produce(Producer<? extends T> p);

Citando Joshua Bloch em Effective Java, 3ª Edição:

Item 31: Use wildcards limitados para aumentar a flexibilidade da API

O uso de tipos wildcard em APIs, embora trabalhoso, torna as APIs muito mais flexíveis. Se você estiver escrevendo uma biblioteca que será amplamente utilizada, o uso adequado de tipos wildcard deve ser considerado obrigatório. Lembre-se da regra básica: producer-extends, consumer-super (PECS). Lembre-se também de que todos os Comparables e Comparators são consumers.

Utilize as opções da inspeção para alternar o relato do seguinte:

  • Classes invariantes. Um exemplo de classe invariante é java.util.List<T>, porque ela tanto aceita valores (através do método List.add(T)) quanto produz valores (através do método T List.get()).

    Por outro lado, uma classe contravariante apenas recebe valores, por exemplo, java.util.function.Consumer<T> com seu único método accept(T). Da mesma forma, uma classe covariante apenas produz valores, por exemplo, java.util.function.Supplier<T> com seu único método T get().

    Frequentemente utiliza-se wildcards limitados em classes covariantes/contravariantes, mas evita-se o uso de wildcards em classes invariantes, como void process(List<? extends T> l). Desativar esta opção permite ignorar tais classes invariantes e manter sua tipagem estrita, como void process(List<T> l).

  • Métodos private que podem ser considerados como não fazendo parte de uma API pública.

  • Métodos de instância.
