Relata a atribuição de serviços de aplicação a campos static final ou propriedades imutáveis.

Campos static final (Java) ou propriedades imutáveis static com backing field (Kotlin)

Nota: No texto a seguir, static em Kotlin refere-se a membros de objetos não anônimos ou declarações de nível superior.

Tais atribuições de serviço levam a um estado global, impossibilitando a remoção de uma aplicação e a configuração de outra em testes; portanto, testes repetidos no mesmo processo podem falhar. A única exceção são chamadas de construtor explícitas usadas para armazenar instâncias dummy/padrão.

A forma recomendada de evitar o armazenamento de serviços é recuperá-los localmente. Como alternativa, eles podem ser encapsulados em um java.util.function.Supplier (Java, Kotlin) ou a propriedade pode ser convertida em uma função (Kotlin).

Exemplo (Java):


// Incorreto:
private static final ManagingFS ourInstance = ApplicationManager.getApplication().getService(ManagingFS.class);


// Correto:
private static final Supplier<ManagingFS> ourInstance = CachedSingletonsRegistry.lazy(() -> {
  return ApplicationManager.getApplication().getService(ManagingFS.class);
});


// Exceção:
private static final UniqueVFilePathBuilder DUMMY_BUILDER = new UniqueVFilePathBuilder()

Recuperação de instâncias de serviço através de propriedades imutáveis static (Kotlin)

Embora a atribuição de um serviço a uma propriedade sem um backing field não cause os problemas mencionados acima, é preferível recuperar serviços usando um método getInstance() explícito em vez de uma propriedade:

  • Isso torna mais claro no call site que a operação pode envolver o carregamento de um serviço, cujo custo pode não ser desprezível.
  • O carregamento de serviços pode lançar exceções, e exceções lançadas por chamadas de método são menos inesperadas do que aquelas causadas pelo acesso a propriedades.
  • O uso (excessivo) de propriedades pode ser propenso a erros, pois uma propriedade pode ser alterada acidentalmente para uma com um inicializador em vez da propriedade correta (porém mais verbosa) com um getter, e tal alteração pode ser facilmente ignorada.
  • O uso de um método em vez de uma propriedade mantém a consistência nas chamadas MyApplicationService.getInstance() ao serem utilizadas tanto em Kotlin quanto em Java.
  • O uso do método torna o MyApplicationService.getInstance() consistente com MyProjectService.getInstance(project), tanto na declaração quanto no call site.

Para um melhor desempenho das ferramentas, recomenda-se sempre manter o tipo de retorno do método explícito.

Exemplo:


@Service
class MyApplicationService {
  companion object {
    @JvmStatic
    val instance: MyApplicationService // Incorreto
       get() = service()
  }
}


@Service
class MyApplicationService {
  companion object {
    @JvmStatic
    fun getInstance(): MyApplicationService = service() // Correto
  }
}
