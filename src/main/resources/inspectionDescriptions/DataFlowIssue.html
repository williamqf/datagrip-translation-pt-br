Com base na análise de fluxo de dados, relata construções de código que sempre violam contratos de nulidade, podem lançar exceções ou são simplesmente redundantes.

Exemplo:

if (array.length < index) {
  System.out.println(array[index]);
} // O índice do array está sempre fora dos limites

if (str == null) System.out.println("str is null");
System.out.println(str.trim());
// A última instrução pode lançar uma NPE

@NotNull
Integer square(@Nullable Integer input) {
    // Viola o contrato do método
    return input == null ? null : input * input;
}

O comportamento da inspeção pode ser controlado por diversas anotações, como as de nulidade, a anotação @Contract, a anotação @Range, entre outras.

Configuração da inspeção:

  • Use a opção Sugerir anotação @Nullable para métodos/campos/parâmetros onde valores anuláveis são usados para emitir avisos quando um valor nulo é passado como argumento para um método com um parâmetro não anotado, armazenado em um campo não anotado ou retornado de um método não anotado. Nesses casos, a inspeção sugerirá a propagação da anotação @Nullable. Você também pode configurar as anotações de nulidade usando o botão Configurar anotações.
  • Use a opção Tratar membros e parâmetros não anotados como @Nullable para assumir que membros não anotados podem ser nulos e, portanto, não devem ser usados em um contexto não nulo.
  • Use a opção Relatar parâmetros que requerem valor não nulo com usos de argumento literal null para relatar parâmetros de método que não podem ser nulos (por exemplo, são desreferenciados imediatamente no corpo do método), mas para os quais existem locais de chamada passando um literal null.
  • Use a opção Relatar métodos anuláveis que sempre retornam um valor não nulo para relatar métodos anotados como @Nullable, mas que sempre retornam valores não nulos. Nesse caso, sugere-se alterar a anotação para @NotNull.
  • Use a opção Ignorar instruções assert para controlar como a inspeção trata instruções assert. Por padrão, a opção está desativada, o que significa que as asserções são tratadas como executadas (modo -ea). Se ativada, as asserções serão completamente ignoradas (modo -da).
  • Use a opção Relatar problemas que ocorrem apenas em alguns caminhos de código para controlar se problemas que podem ocorrer apenas em alguns caminhos de código devem ser relatados. Se esta opção estiver desativada, avisos como exception is possible (exceção é possível) não serão relatados. A inspeção relatará apenas avisos como exception will definitely occur (exceção ocorrerá definitivamente). Este modo pode reduzir drasticamente o número de falsos positivos, especialmente se o código não estiver anotado consistentemente com anotações de nulidade e contrato. É por isso que é útil para encontrar os problemas mais importantes em bases de código legadas.

Antes da versão 2022.3 do IntelliJ IDEA, esta inspeção fazia parte da inspeção "Condições constantes e exceções". Agora, ela foi dividida em duas inspeções: "Valores constantes" e "Nulidade e problemas de fluxo de dados".
