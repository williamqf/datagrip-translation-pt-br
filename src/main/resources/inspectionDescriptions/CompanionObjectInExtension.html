Relata o uso incorreto de objetos acompanhantes (companion objects) em extensões.

Objetos acompanhantes do Kotlin são sempre criados ao tentar carregar a classe que os contém, e as implementações de pontos de extensão devem ter um baixo custo de criação. O carregamento excessivo de classes em plugins é um grande problema para a inicialização do IDE.

Padrão incorreto:


  class KotlinDocumentationProvider : AbstractDocumentationProvider(), ExternalDocumentationProvider {

      companion object {
          private val LOG = Logger.getInstance(KotlinDocumentationProvider::class.java)
          private val javaDocumentProvider = JavaDocumentationProvider()
      }
  }

Aqui, o KotlinDocumentationProvider é uma extensão registrada no plugin.xml: 


  <lang.documentationProvider language="kotlin"
                              implementationClass="org.jetbrains.kotlin.idea.KotlinDocumentationProvider"
                              order="first"/>

Neste exemplo, o JavaDocumentationProvider é carregado do disco assim que alguém chama new KotlinDocumentationProvider().

Objetos acompanhantes do Kotlin em implementações de pontos de extensão devem conter apenas loggers e constantes simples. Outras declarações podem causar carregamento excessivo de classes ou inicialização precoce de recursos pesados (ex: TokenSet, Regex, etc.) ao carregar a classe da extensão. Observe que mesmo declarações marcadas com @JvmStatic geram uma classe extra para o objeto acompanhante, o que pode resultar em cálculos computacionalmente caros.

Essas declarações devem ser armazenadas em um object ou no nível superior (top-level), em vez de em um objeto acompanhante.

Perguntas frequentes (FAQ)

Como reescrever um ConfigurationType de execução?

Mova a declaração para o nível superior (top-level):


  // Correto, usando fun de nível superior
  internal fun mnRunConfigurationType(): MnRunConfigurationType = runConfigurationType<MnRunConfigurationType>()

  internal class MnRunConfigurationType : ConfigurationType {
    companion object { // INCORRETO
      fun getInstance(): MnRunConfigurationType = runConfigurationType<MnRunConfigurationType>()
    }
    ...

Como reescrever um FileType?

Antes:


  internal class SpringBootImportsFileType : LanguageFileType(SPILanguage.INSTANCE, true) {
    companion object {
      val FILE_TYPE = SpringBootImportsFileType()
      ...

Depois:


  internal object SpringBootImportsFileType : LanguageFileType(SPILanguage.INSTANCE, true) {
  ...

No plugin.xml, use o fieldName INSTANCE:


  <fileType name="Spring Boot Imports"
              fieldName="INSTANCE"
              implementationClass="com.intellij.spring.boot.spi.SpringBootImportsFileType"/>

Como reescrever o CounterUsagesCollector?

API Interna

Antes:


  class AntActionsUsagesCollector : CounterUsagesCollector() {
    override fun getGroup(): EventLogGroup = GROUP

    companion object {
      private val GROUP = EventLogGroup("build.ant.actions", 1)

      @JvmField
      val runSelectedBuildAction = GROUP.registerEvent("RunSelectedBuild")
   }
}

Depois:


 object AntActionsUsagesCollector : CounterUsagesCollector() {
  override fun getGroup(): EventLogGroup = GROUP

  private val GROUP = EventLogGroup("build.ant.actions", 1)

  @JvmField
  val runSelectedBuildAction = GROUP.registerEvent("RunSelectedBuild")
}
