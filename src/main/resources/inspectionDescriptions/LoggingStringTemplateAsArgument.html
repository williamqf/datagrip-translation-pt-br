Relata templates de string usados como argumentos para métodos de log do SLF4J e Log4j 2. O método org.apache.logging.log4j.Logger.log() e suas sobrecargas são suportados apenas com a opção Todos os níveis de log. Os templates de string são avaliados em tempo de execução, mesmo quando a mensagem de log não é registrada; isso pode impactar negativamente o desempenho. Recomenda-se o uso de mensagens de log parametrizadas, que não são avaliadas quando o log está desativado.

Exemplo (para Kotlin):


      val variable1 = getVariable()
      logger.info("variable1: $variable1")

Após a aplicação da correção rápida (para Kotlin):


      val variable1 = getVariable()
      logger.info("variable1: {}", variable1)

Observe que a substituição sugerida pode não ser equivalente ao código original, por exemplo, quando o template de string contém chamadas de método ou expressões de atribuição.

  • Use a lista Avisar em para ignorar certos níveis de log mais altos. Níveis de log mais altos podem estar sempre habilitados e os argumentos serão sempre avaliados.
  • Use a opção Não avisar quando apenas expressões com tipos primitivos, seus wrappers ou String estiverem contidos para ignorar templates de string que contenham apenas expressões com tipos primitivos, seus wrappers ou Strings. Por exemplo, isso pode ajudar a evitar o carregamento tardio (lazy loading) de coleções. Note que a criação de uma string em tempo de execução, mesmo usando apenas expressões com tipos primitivos, seus wrappers ou Strings, pode impactar negativamente o desempenho.
  • Use a opção Não avisar para chamadas contendo apenas uma exceção como argumento após o argumento da mensagem para ignorar chamadas que contenham apenas uma exceção como argumento após o argumento da mensagem. Por exemplo, isso pode ser útil ao ajustar o comportamento da inspeção com um estilo de código personalizado.
