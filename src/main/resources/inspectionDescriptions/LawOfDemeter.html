Relata violações da Lei de Demeter.

A Lei de Demeter não é exatamente uma lei, mas especifica uma diretriz de estilo: nunca chame um método em um objeto obtido a partir de outra chamada. O código que segue esta diretriz é mais fácil de manter, ajustar e refatorar, possui menos acoplamento entre métodos, menos duplicidade e melhor ocultação de informações. Por outro lado, pode ser necessário escrever muitos métodos wrapper para satisfazer esta diretriz.

Exemplo:


  boolean pay(Customer c, Invoice invoice) {
    int dollars = c.getWallet().contents; // violação
    if (dollars >= invoice.getAmount()) {
      Wallet w = c.getWallet();
      w.subtract(invoice.getAmount()); // violação
      return true;
    }
    return false;
  }
O exemplo acima poderia ser melhor implementado como um método payInvoice(Invoice invoice) em Customer.

Use a opção Ignorar chamadas a métodos de biblioteca e acessos a campos de biblioteca para ignorar violações da Lei de Demeter que não podem ser corrigidas sem alterar a biblioteca.
