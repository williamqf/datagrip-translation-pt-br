Relata casos em que um objeto não seguro é passado para um método com um parâmetro marcado com a anotação @Untainted, retornado de um método anotado ou atribuído a um campo, parâmetro ou variável local anotado. Os métodos set e get do Kotlin para campos não são suportados como pontos de entrada.

Objetos seguros (na mesma classe) são:

  • Literais de string, instâncias de interface ou objetos enum
  • Resultados de chamadas de métodos marcados como @Untainted
  • Campos private, atribuídos apenas com literais de string e que possuem um inicializador seguro
  • Campos final que possuem um inicializador seguro
  • Variáveis locais ou parâmetros marcados como @Untainted e que não foram atribuídos a partir de um objeto não seguro
Este campo, variável local ou parâmetro não deve ser passado como argumento para um método ou usado como qualificador, ou deve ser um tipo primitivo, seu invólucro (wrapper) ou um objeto imutável. Além disso, campos static final também são considerados seguros.

A análise é executada apenas dentro de um arquivo. Para processar dependências de outras classes, use as opções. A análise se estende a métodos private ou static e possui limites de profundidade de propagação.

Exemplo: 


  void doSmth(boolean b) {
    String s = safe();
    String s1 = "other";
    if (b) s1 = s;
    sink(s);
  }

  String sink(@Untainted String s) {}

Aqui não há atribuição de string não segura para s, portanto nenhum aviso é gerado. Por outro lado: 


  void doSmth(boolean b) {
    String s = safe();
    String s1 = "other";
    s1 = foo();
    if (b) s = s1;
    sink(s);        // Aviso aqui
  }
  
  String foo();

  String sink(@Untainted String s) {}

Aqui há um aviso porque s1 possui um estado desconhecido após a atribuição do resultado da chamada de foo.
