Com base na análise do fluxo de dados, informa construções de código que sempre violam contratos de nulidade, podem lançar exceções ou são apenas redundantes.

Exemplos:

if (array.length < index) {
  System.out.println(array[index]);
} // O índice do array sempre está fora dos limites

if (str == null) System.out.println("str is null");
System.out.println(str.trim());
// A última declaração pode lançar NPE

@NotNull
Integer square(@Nullable Integer input) {
    // Viola o contrato do método
    return input == null ? null : input * input;
}

O comportamento desta inspeção pode ser controlado por várias anotações, como anotações de nulidade, anotações @Contract, anotações @Range etc.

Configurar a inspeção:

  • Use a opção Sugerir anotação @Nullable para métodos/campos/parâmetros que podem ter valor nulo para emitir avisos quando um valor nulo é passado como argumento para um método com parâmetros não anotados, armazenado em um campo não anotado ou retornado por um método não anotado. Nesse caso, esta inspeção irá sugerir a propagação da anotação @Nullable. Você também pode configurar anotações de nulidade usando o botão Configurar anotações.
  • Use a opção Tratar membros e parâmetros não anotados como @Nullable para presumir que os membros não anotados podem ser nulos e, portanto, eles não devem ser usados em contextos não nulos.
  • Use a opção Relatar parâmetros obrigatórios não nulos com uso de argumentos literais nulos para relatar parâmetros de método que não podem ser nulos (por exemplo, são desreferenciados imediatamente no corpo do método) mas que tem pontos de chamada que passam literais null.
  • Use a opção Relatar métodos que podem aceitar valores nulos que sempre retornam valores não nulos para relatar métodos que são anotados com @Nullable, mas que sempre retornam valores não nulos. Recomenda-se alterar a anotação para @NotNull nesses casos.
  • Use a opção Ignorar sentenças assert para controlar como a inspeção lida com sentenças assert. Por padrão, esta opção está desabilitada, o que significa que as asserções são consideradas como sendo executadas (modo -ea). Se esta opção for habilitada, as asserções serão completamente ignoradas (modo -da).
  • Use a opção Relatar problemas que ocorrem apenas em determinados caminhos de código para controlar se problemas que ocorrem apenas em determinados caminhos de código são relatados. Se esta opção for desabilitada, avisos como exception is possible não serão relatados. Esta inspeção irá apenas relatar avisos como exception will definitely occur. Este modo pode reduzir significativamente o número de falsos positivos, especialmente se o código não for anotado consistentemente com anotações de nulidade e contrato. É por isso que ele é útil para encontrar os problemas mais importantes na base de código legado.

Antes do IntelliJ IDEA 2022.3, esta inspeção era parte da inspeção "Condições constantes e excepções". Agora ela foi dividida em duas inspeções: "Valores constantes" e "Problemas de nulidade e fluxo de dados".
