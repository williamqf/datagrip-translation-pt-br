Relata acessos potencialmente problemáticos ao receptor CoroutineScope implícito dentro de um contexto de suspensão.

Quando uma função de suspensão ou lambda captura um receptor CoroutineScope implícito de um contexto externo, isso pode levar a comportamentos inesperados.

Esse código pode violar facilmente as regras de concorrência estruturada, mesmo que pareça estar bem estruturado.

Isso pode resultar em comportamentos incorretos ao lidar com erros, cancelar computações ou gerenciar o tempo de vida.

Um exemplo típico:


fun processFlow(flow: Flow<String>) {
  runBlocking {
    flow.collectLatest {
      
      // Iniciado no CoroutineScope this@runBlocking
      launch { 
        longProcessing(value)
      }
  
    }
  }
}
  
suspend fun longProcessing(value: String) { ... }

No exemplo acima, a chamada launch { ... } inicia uma corrotina no CoroutineScope implícito da chamada runBlocking externa.

Como resultado, o collectLatest não consegue cancelar essas corrotinas quando necessário.

Esta inspeção detectará casos em que launch é chamado em um CoroutineScope capturado de fora do lambda de suspensão atual (ou seja, o corpo do collectLatest).

Para mais informações sobre este problema específico, consulte este issue do GitHub.

Soluções possíveis:

Para resolver esse problema, você pode:

  • Usar o builder coroutineScope { ... } para criar um escopo filho vinculado ao tempo de vida da função de suspensão
  • Reorganizar o código para que o CoroutineScope implícito venha do contexto sintático atual

Se você tiver certeza de que o código está correto, poderá tornar sua intenção explícita especificando o receptor. No exemplo acima, seria this@runBlocking.launch { ... }. Observe que, neste caso, a semântica do código não mudará.
