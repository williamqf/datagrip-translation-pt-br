Relata dados não seguros em expressões EL não escapadas em páginas JSP. Tais casos podem levar a vulnerabilidades de Cross-site scripting (XSS).

A descrição abaixo utiliza o termo dados não contaminados (untainted) para se referir a dados seguros e confiáveis, em oposição a contaminados (tainted - não seguros, não confiáveis). Consulte verificação de contaminação (taint checking) para obter mais informações.

Objetos seguros (não contaminados) são:

  • Tipos de dados primitivos (exceto char)
  • Resultados de chamadas de métodos marcados como untainted
  • Resultados de chamadas de funções de escape
  • Campos marcados como untainted

Para os dados fornecidos, as páginas JSP podem ter contextos diferentes e, se os dados forem seguros para um contexto, isso não significa necessariamente que sejam seguros para todos os contextos. Consulte a Cross-Site Scripting Prevention Cheat Sheet para obter exemplos. Esta inspeção ignora dados marcados como untainted (seguros) em todos os contextos.

Esta inspeção suporta as seguintes opções para marcar métodos e campos como seguros:

  • Usando qualquer anotação da lista configurável de anotações untainted
  • Configurando uma lista de métodos e campos seguros
  • Todos os campos static final podem ser marcados como untainted selecionando a opção "Tratar campos static final como untainted" na guia "Métodos e campos não contaminados"

Esta inspeção não destaca locais onde dados não seguros alcançam uma marcação de segurança em páginas JSP. Por exemplo, a tag out da taglib JSTL é segura porque escapa o valor fornecido por padrão. A lista de marcações de segurança é configurável (omitir o valor de tag ou attribute permite definir todo o escopo).

A correção rápida "Envolver com a tag JSTL <out>" envolve a expressão EL com a tag JSTL <out>.

Antes:

  
    ${foo}

Depois:

  
  <%@ taglib prefix="c" uri="http://java.sun.com/jsp/jstl/core" %>
<c:out value="${foo}"/>

A correção "Envolver com a função JSTL escapeXml()" envolve valores não seguros com a função de escape escapeXml.

Antes:

  
  <%@ taglib prefix="custom" uri="WEB_INF/custom.tld" %>
  <custom:tag value="${foo}"/>

Depois:

  
    <%@ taglib prefix="custom" uri="WEB_INF/custom.tld" %>
    <%@ taglib prefix="fn" uri="http://java.sun.com/jsp/jstl/functions" %>
    <custom:tag value="${fn:escapeXml(foo)}"/>

Existem mais correções rápidas para adicionar a tag selecionada às tags seguras ou para anotar métodos (ou campos) relevantes com uma anotação untainted.

Funções de escape personalizadas podem ser adicionadas (consulte a guia "Funções de escape"). Atualmente, ela contém funções do JSTL que retornam valores seguros (por exemplo, a conhecida função escapeXml). Não é necessário definir funções JSTL duas vezes usando URIs de namespace diferentes. Se um entre http://java.sun.com/jsp/jstl/functions ou http://java.sun.com/jstl/functions for usado, o outro também será verificado. Também é possível ignorar funções de escape em qualquer lugar com base no nome. Para fazer isso, deixe o campo URI do namespace vazio.
