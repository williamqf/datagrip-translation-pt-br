Relata o código que nunca é alcançado com base na análise de fluxo de dados. Pode ser o resultado de uma condição anterior que é sempre true ou sempre false, um corpo de loop inacessível ou uma seção catch. Geralmente (embora nem sempre), o código inacessível é o resultado de avisos anteriores, portanto, verifique os avisos das inspeções "Problemas de nulidade e fluxo de dados", "Valores constantes" ou "Operações redundantes em contêineres vazios" para entender melhor a causa.

Exemplo:


  void finishApplication() {
    System.exit(0);
    System.out.println("Application is terminated"); // código inacessível
  }

Observe que esta inspeção depende da inferência de contratos de método. Em particular, se você chamar um método static ou final que sempre lança uma exceção, o contrato "sempre falha" será inferido e o código após a chamada do método será considerado inacessível. Exemplo: 


  void run() {
    performAction();
    System.out.println("Action is performed"); // código inacessível
  }
  
  static void performAction() {
    throw new AssertionError();
  }

Isso pode levar a falsos positivos se qualquer tipo de pós-processamento de código for usado, por exemplo, se um processador de anotações posteriormente substituir o corpo do método por algo útil. Para evitar avisos de falso positivo, suprima a inferência automática de contrato usando a anotação explícita @org.jetbrains.annotations.Contract do pacote org.jetbrains:annotations: 


  void run() {
    performAction();
    System.out.println("Action is performed"); // não há mais aviso
  }

  @Contract("-> _") // a implementação será substituída
  static void performAction() {
    throw new AssertionError();
  }
