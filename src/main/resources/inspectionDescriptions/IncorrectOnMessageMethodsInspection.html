Relata métodos WebSocket incorretos anotados com @OnMessage.

Esta anotação marca um método que processa mensagens WebSocket recebidas. Cada ponto de extremidade (endpoint) WebSocket pode ter apenas um método de tratamento de mensagens para cada formato de mensagem WebSocket (texto, binário, pong). Métodos anotados com @OnMessage podem ter o seguinte conjunto de parâmetros em qualquer ordem:

  • Pode conter um dos seguintes conjuntos de parâmetros:
    • Se o método estiver processando uma mensagem de texto:
      • String para receber a mensagem completa
      • Um tipo primitivo Java ou classe equivalente para receber a mensagem completa convertida para esse tipo
      • Um par de String e booleano para receber a mensagem em partes
      • Reader para receber a mensagem completa como um stream de bloqueio
      • Qualquer parâmetro de objeto para o qual o ponto de extremidade tenha um decodificador de texto (Decoder.Text ou Decoder.TextStream)
    • Se o método estiver processando uma mensagem binária:
      • byte[] ou ByteBuffer para receber a mensagem completa
      • Um par de byte[] e booleano, ou ByteBuffer e booleano para receber a mensagem em partes
      • InputStream para receber a mensagem completa como um stream de bloqueio
      • Qualquer parâmetro de objeto para o qual o ponto de extremidade tenha um decodificador binário (Decoder.Binary ou Decoder.BinaryStream)
    • Se o método estiver processando uma mensagem pong:
      • PongMessage para tratar a mensagem pong
  • Qualquer número de parâmetros String ou primitivos Java anotados com @PathParam para pontos de extremidade do servidor
  • Um parâmetro Session opcional

Se o método retornar um valor, o runtime do WebSocket o interpretará como uma mensagem a ser retornada ao par (peer). Os tipos de retorno permitidos são String, ByteBuffer, byte[], qualquer primitivo Java ou classe equivalente, e qualquer tipo com um codificador (encoder). Se o método retornar um valor primitivo Java, a implementação deverá construir a mensagem de texto a ser enviada usando a representação de string Java padrão do primitivo Java, a menos que o desenvolvedor tenha fornecido um codificador para o tipo configurado para este ponto de extremidade, caso em que esse codificador deve ser usado. Se o método retornar uma classe equivalente a um primitivo Java, a implementação deverá construir a mensagem de texto a partir do equivalente primitivo conforme descrito anteriormente.
